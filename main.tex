\section{Introduction}
\label{sec:introduction}

This document defines the shared architecture, specifications, and communication protocols for the Advanced Programming course project. Its purpose is to enable independent groups to develop compatible components by providing:

\begin{itemize}[leftmargin=*]
    \item \important{Consistent interfaces} for all game entities
    \item \important{Unambiguous protocols} for inter-component communication
    \item \important{Rigorous specifications} of component behavior
    \item \important{Safety guarantees} through user-inconstructible types
    \item \important{Compatibility standards} for cross-group integration
\end{itemize}

\note{This documentation serves as the authoritative reference for external implementers who need to create planets, explorers, or other components that interoperate with existing systems. The focus is on public interfaces and protocols, not internal implementations.}

\section{System Architecture}
\label{sec:architecture}

\subsection{Crate Structure}
The common crate provides the foundation for all implementations:

\begin{verbatim}
common/
├── src/
│   ├── lib.rs             # Public exports
│   ├── components/        # Game entity definitions
│   │   ├── asteroid.rs     # Asteroid struct
│   │   ├── energy_cell.rs  # EnergyCell with charging logic
│   │   ├── forge.rs        # Singleton Forge
│   │   ├── planet.rs       # Planet AI trait & types
│   │   ├── resource.rs     # Resource types & recipes
│   │   ├── rocket.rs       # Rocket struct
│   │   └── sunray.rs       # Sunray struct
│   └── protocols/         # Communication protocols
│       ├── messages.rs    # All message enums
│   ├── protocols.rs
│   ├── components.rs
│   ├── logging.rs
└── Cargo.toml
\end{verbatim}

\newpage

\subsection{Component Relationships}
The system follows a strict hierarchical architecture:

\begin{itemize}
    \item \important{Orchestrator}: Central coordinator owning a singleton Forge
    \item \important{Planets}: Autonomous entities with AI behavior and resource capabilities
    \item \important{Explorers}: Mobile agents that traverse between planets
    \item \important{Forge}: Singleton component creating Sunrays and Asteroids
    \item \important{Communication}: All interactions via defined message protocols
\end{itemize}

\subsection{Guiding Principles}
\begin{itemize}[leftmargin=*]
    \item \important{Protocol-first design}: Interfaces defined by message protocols
    \item \important{User-inconstructible types}: Critical types cannot be created by users
    \item \important{No undocumented panics}: All failure modes are specified
    \item \important{Clippy-pedantic compliance}: Highest code quality standards
    \item \important{No unsafe code}: Memory safety guaranteed
\end{itemize}

\section{Core Components}
\label{sec:components}

\subsection{Forge}
\label{subsec:forge}

\note{\faLightbulb\ \textbf{Purpose:} Singleton component that creates Sunrays and Asteroids. Ensures system-wide uniqueness.}

\subsubsection{Interface}
\begin{lstlisting}[style=rustcool, caption={Forge Public Interface}, label=lst:forge-interface]
/// Singleton that creates Sunrays and Asteroids.
/// Construction fails if a Forge already exists.
pub struct Forge {
    _private: (),
}

impl Forge {
    /// Creates the singleton Forge.
    /// Returns Err if Forge already exists.
    pub fn new() -> Result<Forge, String>;
    
    /// Creates a new Sunray.
    pub fn generate_sunray(&self) -> Sunray;
    
    /// Creates a new Asteroid.
    pub fn generate_asteroid(&self) -> Asteroid;
}
\end{lstlisting}

\subsubsection{Protocol Rules}
\begin{itemize}
    \item \important{Singleton enforcement}: Only one Forge can exist in the entire system
    \item \important{Orchestrator requirement}: Every orchestrator must own a Forge (not borrow)
    \item \important{Exclusive creation}: Only Forge can create Sunrays and Asteroids
\end{itemize}

\warningbox{\textbf{Critical Requirement:} Orchestrator implementations must contain a \code{Forge} field (not a reference) to prevent multiple orchestrators. This is enforced at runtime.}

\subsubsection{Source Reference}
Complete implementation: \href{https://unitn-ap-2025.github.io/common-docs/src/common_game/protocols/orchestrator.rs.html}{forge.rs}




\subsection{Rocket}
\label{subsec:rocket}

\note{\faLightbulb\ \textbf{Purpose:} Planetary defense mechanism against Asteroids.}

\subsubsection{Interface}
\begin{lstlisting}[style=rustcool, caption={Rocket Construction Interface}, label=lst:rocket-interface]
/// Planetary defense against Asteroids.
/// Built using charged EnergyCells.
pub struct Rocket {
    _private: (),
}

impl Rocket {
    /// Constructs Rocket from charged EnergyCell.
    /// Consumes the cell's energy.
    pub fn new(energy_cell: &mut EnergyCell) -> Result<Rocket, String>;
}
\end{lstlisting}

\subsubsection{Construction Protocol}
\begin{itemize}
    \item \important{Input}: One fully charged EnergyCell
    \item \important{Consumption}: Cell is discharged during construction
    \item \important{Availability}: Only planet types A and C can have rockets
    \item \important{Error case}: Construction fails with \code{Err} if EnergyCell is not charged
\end{itemize}

\note{\faLightbulb\ \textbf{Deflection Mechanism:} Rockets have no inherent behavior. Asteroid deflection is handled by the Planet's \code{handle\_asteroid()} method, which returns an owned Rocket if available for deflection.}

\subsubsection{Source Reference}
Complete implementation: \href{https://unitn-ap-2025.github.io/common-docs/src/common_game/components/rocket.rs.html}{rocket.rs}

\subsection{Sunray}
\label{subsec:sunray}

\note{\faLightbulb\ \textbf{Purpose:} Immutable energy unit used to charge EnergyCells. Created exclusively by the Generator.}

\subsubsection{Interface}
\begin{lstlisting}[style=rustcool, caption={Sunray Structure}, label=lst:sunray-struct]
/// Immutable energy unit.
/// Created only by Generator, consumed by EnergyCells.
pub struct Sunray {
    _private: (),
}
\end{lstlisting}

\subsubsection{Rules}
\begin{itemize}
    \item \important{Single-use}: Each Sunray charges exactly one EnergyCell
    \item \important{Consumption}: Sunray is consumed during charging
    \item \important{Idempotent charging}: Charging already-charged cell wastes Sunray
\end{itemize}

\subsubsection{Source Reference}
Complete implementation: \href{https://unitn-ap-2025.github.io/common-docs/src/common_game/components/sunray.rs.html}{sunray.rs}

\subsection{Asteroid}
\label{subsec:asteroid}

\note{\faLightbulb\ \textbf{Purpose:} Destructive object that planets must deflect using Rockets.}

\subsubsection{Interface}
\begin{lstlisting}[style=rustcool, caption={Asteroid Structure}, label=lst:asteroid-struct]
/// Destructive object targeting planets.
/// Created only by Generator, deflected by Rockets.
pub struct Asteroid {
    _private: (),
}
\end{lstlisting}

\subsubsection{Rules}
\begin{itemize}
    \item \important{Rocket required}: Planet without Rocket is destroyed
    \item \important{Rocket consumption}: Rocket is consumed during deflection
    \item \important{Immediate response}: Planet must respond with \code{AsteroidAck} (with \code{Some(Rocket)}) if defended, or \code{KillPlanetAck} if destroyed
    \item \important{Termination}: Planet execution terminates after sending \code{KillPlanetAck}
    \item \important{Explorer termination cascade}: All Explorers currently present on the destroyed planet must also be terminated
\end{itemize}

\subsubsection{Source Reference}
Complete implementation: \href{https://unitn-ap-2025.github.io/common-docs/src/common_game/components/asteroid.rs.html}{asteroid.rs}

\subsection{EnergyCell}
\label{subsec:energy-cell}

\note{\faLightbulb\ \textbf{Purpose:} Stores energy for resource generation, combination, and rocket construction.}

\subsubsection{Interface}
\begin{lstlisting}[style=rustcool, caption={EnergyCell Public Interface}, label=lst:energy-cell-interface]
/// Stores binary energy state (charged/uncharged).
pub struct EnergyCell {
    charge: bool,
}

impl EnergyCell {
    /// Creates new uncharged cell.
    pub fn new() -> Self;
    
    /// Charges cell using Sunray.
    /// Wastes Sunray if already charged.
    pub fn charge(&mut self, sunray: Sunray);
    
    /// Discharges cell for resource operations.
    /// Returns Err if cell not charged.
    pub fn discharge(&mut self) -> Result<(), String>;
    
    /// Checks if cell is charged.
    pub fn is_charged(&self) -> bool;
}
\end{lstlisting}

\begin{table}[H]

\textbf{Operation} & \textbf{Energy Cells} & \textbf{Effect} \\
\hline
Basic Resource Generation & 1 charged cell & Cell discharged \\
Complex Resource Combination & 1 charged cell & Cell discharged \\
Rocket Construction & 1 charged cell & Cell discharged \\
Sunray Charging & 1 Sunray & Cell becomes charged \\
\hline

\subsubsection{Key Constraints}
\begin{itemize}
    \item \important{Binary state}: Charge is either 0 or 1 (maximum charge is one)
    \item \important{Initial state}: New cells start uncharged (zero initial charge)
    \item \important{Idempotent charging}: Charging already-charged cell wastes Sunray
    \item \important{Discharge failure}: Attempting to discharge uncharged cell returns \code{Err("EnergyCell not charged!")}
\end{itemize}

\subsubsection{Planet AI Trait}

\begin{lstlisting}[style=rustcool, caption={PlanetAI Trait}, label=lst:planet-ai-trait]
/// Defines planet behavior through message handlers.
pub trait PlanetAI: Send {
    /// Handles orchestrator messages (except Start/Stop/Asteroid).
    fn handle_orchestrator_msg(
        &mut self,
        state: &mut PlanetState,
        generator: &Generator,
        combinator: &Combinator,
        msg: OrchestratorToPlanet,
    ) -> Option<PlanetToOrchestrator>;
    
    /// Handles all explorer messages.
    fn handle_explorer_msg(
        &mut self,
        state: &mut PlanetState,
        generator: &Generator,
        combinator: &Combinator,
        msg: ExplorerToPlanet,
    ) -> Option<PlanetToExplorer>;
    
    /// Handles asteroid threats.
    /// Returns Some(Rocket) to survive, None to be destroyed.
    fn handle_asteroid(
        &mut self,
        state: &mut PlanetState,
        generator: &Generator,
        combinator: &Combinator,
    ) -> Option<Rocket>;
    
    /// Called when planet AI starts.
    fn start(&mut self, state: &PlanetState);
    
    /// Called when planet AI stops.
    fn stop(&mut self, state: &PlanetState);
}
\end{lstlisting}

\note{\faLightbulb\ \textbf{Important:} The \code{Generator} and \code{Combinator} parameters refer to \code{resource::Generator} and \code{resource::Combinator} (resource recipe managers), not \code{components::generator::Generator} (Sunray/Asteroid creator).}

pub struct PlanetState {

    /// Returns mutable borrow of first charged cell with index.

    /// Called by orchestrator in separate thread.

        // 2. Call ai.start()
        // 3. Poll orchestrator and explorer channels using crossbeam_channel

    }
}

\subsection{Explorer}
\label{subsec:explorer}

\subsubsection{Core Responsibilities}
\begin{itemize}
    \item \important{Resource collection}: Gathers basic and complex resources
    \item \important{Planet navigation}: Moves between connected planets
    \item \textbf{Resource management}: Maintains inventory of collected resources
    \item \textbf{Protocol compliance}: Follows defined message protocols
\end{itemize}

\subsubsection{Resource Storage}
\note{\faLightbulb\ \textbf{Design Note:} Explorers manage resources internally without a formal Bag trait. Resource storage implementation is left to individual explorer implementations, but must support the required resource operations.}

\begin{itemize}
    \item \faBook\ \textbf{Reference:} \code{main.pdf 2.7.2}: "The bag is the set of resources... The bag could be, for example, a \code{Vec<BasicResource>} and a \code{Vec<ComplexResource>}"
    \item \faCode\ \textbf{Implementation:} Each developer team designs their own storage solution within the provided protocol constraints
    \item \faLock\ \textbf{Constraint:} The bag cannot contain energy cells, which are local to a Planet and cannot be moved 
\end{itemize}

\subsubsection{Key Constraints}
\begin{itemize}
    \item \important{No energy creation}: Cannot create or modify EnergyCells
    \item \important{Channel-based communication}: Uses provided channels for all interactions
    \item \important{Orchestrator-controlled movement}: Movement initiated through orchestrator
    \item \important{Resource transfer}: Resources move with explorer during planet transitions
    \item \important{No explorer interaction}: An Explorer cannot initiate interactions with other Explorers
\end{itemize}

\warningbox{\faUsers\ \textbf{Isolation Principle:} Explorers operate independently and cannot directly communicate with each other. All coordination must happen through the orchestrator or via shared planetary systems.}

\subsubsection{Message Flow Rules}
\begin{itemize}
    \item Planets in stopped state respond with \code{Stopped} to any message except \code{StartPlanetAI} or \code{KillPlanet}
    \item \code{KillPlanet} terminates planet execution regardless of state
    \item After sending \code{KillPlanetAck}, planet execution terminates
    \item \code{AsteroidAck} with \code{None} rocket indicates planet cannot defend
    \item \important{Planet destruction cascade}: When a planet is destroyed (sends \code{KillPlanetAck} in response to asteroid), all Explorers currently on that planet must also be terminated
\end{itemize}

\section{Observability and Diagnostics}
\label{sec:observability}

\subsection{Unified Logging Framework}
All system components use a common structured logging framework for monitoring and debugging. This provides consistent, machine-readable logs across planets, explorers, and orchestrators.

\note{\faLightbulb\ The logging system enables cross-component debugging by standardizing log format and metadata. Implementers can trace message flows and diagnose issues without coordinating log formats with other teams.}


\subsection{Log Structure and Types}
Logs capture comprehensive metadata about system events:

\begin{lstlisting}[style=rustcool, caption={Core Logging Types}]
/// Identifies component types in the system
pub enum ActorType {
    Planet, Explorer, Orchestrator, User, Broadcast, SelfActor
}

/// Standard log severity levels
pub enum Channel { Error, Warning, Info, Debug, Trace }

/// Categories of system events
pub enum EventType {
    // Inter-component messages
    MessagePlanetToOrchestrator, MessagePlanetToExplorer,
    MessageOrchestratorToPlanet, MessageOrchestratorToExplorer,
    MessageExplorerToPlanet, MessageExplorerToOrchestrator,
    
    // Internal component operations  
    InternalPlanetAction, InternalExplorerAction, 
    InternalOrchestratorAction,
    
    // External interactions
    UserToPlanet, UserToExplorer, UserToOrchestrator
}

/// Structured log entry with full context
pub struct LogEvent {
    pub timestamp_unix: i64,
    pub sender_type: ActorType,
    pub sender_id: u64,
    pub receiver_type: ActorType,
    pub receiver_id: String,
    pub event_type: EventType,
    pub channel: Channel,
    pub payload: BTreeMap<String, String>,
}
\end{lstlisting}

\note{\faLightbulb\ Each log includes sender, receiver, timestamp, and event type. This allows reconstructing complete interaction sequences across the distributed system.}

\subsection{Integration with Rust's Log Ecosystem}
The system integrates with the standard Rust \code{log} crate, allowing implementers to use familiar logging backends like \code{env\_logger}, \code{simple\_logger}, or \code{fern}.

\note{\faLightbulb\ Components emit logs through the \code{log} crate interface. To see output, the final binary must initialize a logger backend. Without initialization, logs are silently dropped.}

\subsection{Logging Requirements for Implementers}
\begin{itemize}
    \item \important{Use Standard Types}: Employ \code{ActorType}, \code{Channel}, \code{EventType} for consistency
    \item \important{Include Full Context}: Always specify sender, receiver, and event category
    \item \important{Choose Appropriate Severity}: Use \code{Error} for failures, \code{Info} for normal operations
    \item \important{Store Variable Data in Payload}: Use the key-value map for dynamic content
\end{itemize}

\warningbox{Logging requires external initialization. Components cannot assume logs will be captured without a logger backend like \code{env\_logger} being configured in the final binary.}

\section{Protocol Summary}
\label{sec:protocol-summary}

\subsection{Key Invariants}
\begin{itemize}
    \item \important{Singleton Generator}: Only one Generator exists system-wide
    \item \important{Energy conservation}: Operations consume charged EnergyCells

    \item \important{Timely responses}: Components respect protocol timeouts
    \item \important{State consistency}: Components maintain valid states
\end{itemize}

\subsection{Error Handling Philosophy}
\begin{itemize}
    \item \important{Recoverable errors}: Return resources when possible
    \item \important{Clean failures}: No panics, proper error reporting
    \item \important{State preservation}: Failed operations don't corrupt state
    \item \important{Resource return}: Failed combinations return inputs
\end{itemize}

\subsection{Extension Points}
The system is designed for extension:
\begin{itemize}
    \item New planet types can implement PlanetAI trait
    \item New resources can be added to resource system
    \item New protocols can extend message enums
    \item Custom explorers can implement movement logic
    \item Additional components can integrate via channels
\end{itemize}
